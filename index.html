Topic 1.1 — What is Claude Code (and what it is not)

What it is

Claude Code is an artificial intelligence agent integrated into your code editor. It understands the context of your project, reads files, writes code, fixes errors, generates documentation and much more — all within VS Code, without you needing to switch between windows or copy and paste content.

Think of it as a senior co-pilot who sits beside you while you work. It sees the same screen as you, knows the open files and responds to your questions based on what is happening in the project at that moment.

What it is not

  • Not an infallible oracle. It makes mistakes, especially with insufficient context. Your role is to review and validate the responses.
  • Does not replace human reasoning. It accelerates execution, but business and architecture decisions remain yours.
  • Not the Claude.ai chat. The experience here is integrated into the editor — much more powerful than an isolated chat window.

Corporate analogy

If Claude.ai is like sending an email to an external consultant asking for help, Claude Code is having that consultant sitting next to you at the desk, with access to the same documents you are editing.

Key Points 1.1

  • Claude Code is an agent integrated into VS Code, not an external chat
  • It operates within the context of your project and its files
  • It is an acceleration tool — you still lead the decisions

Topic 1.2 — Installing VS Code

Why VS Code

Visual Studio Code is the most widely used code editor in the corporate world. It is free, lightweight, extensible and developed by Microsoft with active support. The entire Claude Code experience in this course is designed to work within it.

Step-by-step installation

Step 1 — Download the installer

Visit the official website: https://code.visualstudio.com

Click the download button corresponding to your operating system (Windows, macOS or Linux). The site detects automatically.

Step 2 — Run the installer (Windows)

  1. Open the downloaded .exe file
  2. Accept the license agreement
  3. On the additional options screen, check the following options:
    • ✅ Add to PATH (important for terminal usage)
    • ✅ Register Code as editor for supported file types
    • ✅ Add "Open with Code" action to context menu
  4. Click Install and wait

Step 3 — Run the installer (macOS)

  1. Open the downloaded .dmg file
  2. Drag the VS Code icon to the Applications folder
  3. On first launch, right-click > Open (to bypass macOS security warning)
  4. To add to PATH: open VS Code, press Ctrl+Shift+P, type "Shell Command" and select "Install 'code' command in PATH"

Step 4 — Verify the installation

Open VS Code. You will see the welcome screen. If you got here, everything is fine.

Topic 1.3 — Installing the Claude Code Extension

What is a VS Code extension

Extensions are like apps within VS Code — they add new features without needing to install separate programs. Claude Code comes to you as an official Anthropic extension.

Extension installation

Method 1 — Via Marketplace (recommended)

  1. In VS Code, click the extensions icon in the sidebar (or press Ctrl+Shift+X)
  2. In the search bar, type: Claude Code
  3. Find the extension published by Anthropic (verify the publisher name)
  4. Click Install
  5. Wait for the download and automatic activation

Method 2 — Via Command Palette

  1. Press Ctrl+Shift+P (opens the Command Palette)
  2. Type: Extensions: Install Extensions
  3. Search for Claude Code and install

Confirming the installation

After installing, you will see:

  • A new icon in the activity bar (usually a circular or agent icon)
  • The extension listed as "active" in the installed extensions section
  • A notification asking for authentication (covered in the next topic)

Tip: If the icon does not appear immediately, reload VS Code with Ctrl+Shift+P > "Reload Window".

Key Points 1.3

  • The Claude Code extension is installed directly from the VS Code Marketplace
  • Always verify that the publisher is Anthropic
  • If something does not appear after installing, reloading the window resolves it in most cases

Topic 1.4 — Authentication and Initial Configuration

Why we need to authenticate

Claude Code connects to Anthropic's language models over the internet. Authentication ensures the connection is made with your account, within the limits of your plan, securely.

Authentication process

Step 1 — Open the Claude Code panel

Click the Claude Code icon in the activity bar. A panel will open on the left side or in a dedicated tab.

Step 2 — Start authentication

You will see a button or link to "Sign in" or "Authenticate". Upon clicking, one of two things will happen:

  • Option A: VS Code will open a browser window redirecting to claude.ai — log in with your Anthropic account and authorize access
  • Option B: A verification code will be displayed — you paste that code on the indicated site to complete authentication

Step 3 — Confirm connection

After authenticating, the Claude Code panel will display an active chat interface, indicating that the connection was established successfully.

What to do if authentication fails

Problem Solution
Browser page does not open Check if there is a pop-up blocker in the browser
Error of "invalid token" Try logging out and back in on the extension
Extension does not respond Reload VS Code (Ctrl+Shift+P > Reload Window)

Note about corporate environments: Some companies have proxies or firewalls that block external connections. If you are in a restricted corporate environment, consult your IT team to whitelist the domain *.anthropic.com.

Key Points 1.4

  • Authentication is done once — after that, the extension remains connected
  • In corporate networks, it may be necessary to whitelist the domain *.anthropic.com in the firewall
  • If something fails, reloading VS Code is the first diagnostic step

Topic 1.5 — Navigating the Claude Code Interface

The three ways to interact

In VS Code with Claude Code installed, you have three main forms of interaction:

1. Side panel (Chat)

The chat panel is the main interface. It works like a conversation:

  • You type a message in the lower text box
  • Press Enter to send
  • The agent responds in the upper area

2. Quick commands in the editor

With a file open, you can:

  • Select a code snippet
  • Press Ctrl+Shift+P and search for Claude Code actions
  • Or use configured shortcuts for common actions (explain, refactor, etc.)

3. Inline reference

In some contexts, the agent can suggest completions directly as you type. This depends on the extension configuration.

Anatomy of the chat panel

│  CONVERSATION HISTORY            │
│  ┌──────────────────────────┐   │
│  │ You: [your message]      │   │
│  └──────────────────────────┘   │
│  ┌──────────────────────────┐   │
│  │ Agent: [response]        │   │
│  └──────────────────────────┘   │
│                                  │
│  ──────────────────────────────  │
│  [Type your message...     ] ↵  │
└──────────────────────────────────┘

Context actions

Within the chat, you can:

  • Reference files: Drag a file to the chat, or use @ followed by the file name (depending on the version)
  • Include code: Paste code snippets directly in the message
  • Continue the conversation: The agent maintains memory within the same chat session

Starting a new conversation

To start a conversation without the previous history:

  • Look for the "New Chat" button at the top of the panel
  • Or close and reopen the panel

Best practices: Start a new conversation for each distinct task. This prevents the history of one task from confusing the context of another.

Key Points 1.5

  • The chat panel is the main interface — it works like a continuous conversation
  • The conversation maintains context within the same session
  • Start a new conversation for each distinct task to keep the context clean

Topic 1.6 — Structure of a Conversation with the Agent

How the agent "thinks"

Before understanding how to converse well with the agent, it is useful to understand how it processes information. Claude Code is not a search engine — it does not "search" answers in a database. It reasons from the context you provide.

This means:

  • The better the context, the better the response
  • It does not know what you did not say
  • It builds each response based on everything said in the current conversation

Anatomy of a good prompt (introduction)

A prompt is the message you send to the agent. In this module, we will cover only the essentials — Module 2 covers this topic in full depth.

The minimum for a good prompt:

[WHAT YOU WANT] + [NECESSARY CONTEXT]

Bad example:

"Fix this."

The agent does not know what "this" is, what is wrong, or what "fixed" would look like.

Good example:

"I have a Python function that should calculate the average of a list of numbers, but it is returning zero. Here is the code: [code]. What might be wrong?"

Now the agent has: what you want (error diagnosis), the context (Python function, expected behavior, current behavior) and the concrete material (the code).

Iteration is natural

Rarely is the first response the perfect final answer. Conversation with the agent is iterative — like an alignment meeting, not a single-request form.

If the response was not what you needed:

  • Ask the agent to refine
  • Add more context
  • Rephrase the question from another perspective

Key Points 1.6

  • The agent reasons from context — the more context, the better the response
  • The basic structure of a prompt: [what you want] + [necessary context]
  • Iteration is normal and expected — few conversations end on the first message

Topic 2.1 — Anatomy of an Efficient Prompt

What a prompt really is

A prompt is not just "a question for the agent". It is a work instruction. Think about how you would delegate a task to a new team member — someone competent, but who does not know your project, your conventions, or the context of the situation.

You would not just say "fix the code". You would say: "I need you to review the calcularDesconto function in the pedidos.js file. It is returning negative values when the discount is greater than 100%. It uses TypeScript, so the types need to be correct."

This mindset of clear delegation is what transforms mediocre prompts into powerful prompts.

The 5 elements of a complete prompt

Every efficient prompt contains, at minimum, the relevant elements from this list:

┌─────────────────────────────────────────────────────┐
│  1. ROLE         Who do you want the agent to be?     │
│  2. TASK         What do you want it to do?           │
│  3. CONTEXT      What does it need to know?           │
│  4. FORMAT       How should the response be?          │
│  5. RESTRICTIONS What should it NOT do?               │
└─────────────────────────────────────────────────────┘

Not every prompt needs all 5 elements — it depends on the complexity of the task. But the more complex, the more elements you should include.

Comparative examples

Task: Asking for help writing an email

Element Bad Prompt Good Prompt
Role (absent) "Act as a corporate technical writer"
Task "Write an email" "Write an email informing of a 2-day delay in the report delivery"
Context (absent) "The recipient is my director, they are formal and direct"
Format (absent) "Maximum 5 lines, no attachments mentioned"
Restrictions (absent) "Do not use excessively apologetic language"

Result of the bad prompt:

A generic email that you will throw away.

Result of the good prompt:

An email that you will copy, adjust the name and send in 30 seconds.

Key Points 2.1

  • A prompt is a work instruction, not just a question
  • The 5 elements: Role, Task, Context, Format, Restrictions
  • The more complex the task, the more elements are needed
  • The "clear delegation" mindset is the correct mental model

Topic 2.2 — Clear vs. Vague Instructions

The problem of ambiguity

Claude Code is extremely capable — but it is not a mind reader. When an instruction is ambiguous, it does the only possible thing: assumes. And its assumptions may not match yours.

See this real example of ambiguity:

Prompt: "Improve this code."

What does "improve" mean? It could be:

  • Make it more readable
  • Make it faster
  • Reduce lines
  • Add error handling
  • Add comments
  • Refactor to another pattern

The agent will choose an interpretation. It may get it right. It may get it completely wrong.

The 4 types of vagueness (and how to fix each one)

Type 1 — Vague objective

❌ "Improve this code."
✅ "Refactor this code to improve readability.
    Keep the functionality exactly the same."

Type 2 — Vague scope

❌ "Review the project."
✅ "Review only the file 'auth.js', focusing on the
    token validation functions."

Type 3 — Vague quality criterion

❌ "Make it a professional text."
✅ "Write in a formal tone, no slang, suitable for
    presentation to directors. Avoid technical jargon."

Type 4 — Vague output format

❌ "Explain how it works."
✅ "Explain in at most 5 bullet points, using language
    accessible to non-developers."

Mental exercise: The 3-reader test

Before sending a prompt, ask yourself:

"If 3 different people read this instruction, would they all reach the same conclusion about what to do?"

If the answer is no — add more specificity.

Key Points 2.2

  • Ambiguity makes the agent guess — and it may guess wrong
  • There are 4 types of vagueness: objective, scope, quality criterion and format
  • Use the "3-reader test" to check clarity before sending

Topic 2.3 — The Role of Context

Why context changes everything

Imagine you need medical help. Would you prefer to speak with a general practitioner who has no information about you, or with a specialist who already has your complete history?

Context in Claude Code works the same way. The agent responds differently depending on what it knows about:

  • The project you are working on
  • Your team's conventions
  • The technical level of the target audience
  • The current state of the code or document
  • What you already tried and did not work

Types of context and how to provide them

Project context

Briefly explain what the project is, its purpose and its technology stack:

"I am working on a REST API in Node.js with Express
that manages employee registration for HR. We use
PostgreSQL as the database and JWT for authentication."

Situation context

Explain the current problem with specificity:

"The POST /employees route is returning 500 when the
'cpf' field already exists in the database. It should return 409
with a friendly error message."

Previous attempt context

If you already tried something, say so. This prevents the agent from suggesting what you already know does not work:

"I already tried adding a try/catch around the query,
but the error is not being caught. The problem seems
to be before the query is executed."

Technical restriction context

Limitations the agent needs to respect:

"I cannot use external libraries — only native
Node.js modules in this function."

The rule of minimum sufficient context

More context is not always better. Irrelevant context pollutes the conversation and may confuse the agent.

The rule is: include only the context that directly impacts the response you want.

TOO LITTLE:   "There is a bug in the code."
EXCESSIVE:    [500 lines of code without saying what is wrong]
IDEAL:        Specific function + expected behavior +
              current behavior + what was already tried

Key Points 2.3

  • Context is the difference between a generic response and a precise response
  • Four types: project, situation, previous attempt and technical restriction
  • Use minimum sufficient context — relevant, not excessive

Topic 2.4 — Positive and Negative Examples

Why examples work so well

When you say "write a professional text", the agent has a general idea of what that means. When you show an example of professional text, it knows exactly.

Examples are the most efficient way to communicate quality, style and format expectations — especially for creative or subjective tasks.

Positive examples — "do it like this"

Use when you know exactly how you want the output to be:

"Write commit messages in the style of the examples below:

Good: 'fix: fix CPF validation to accept formatting with dots'
Good: 'feat: add CSV export endpoint for monthly report'
Good: 'refactor: extract authentication logic to separate middleware'

Now write a commit message for the change where I modified
the progressive discount calculation function."

Negative examples — "do not do it like this"

Use when you want to avoid a specific pattern the agent tends to use:

"Write a code review comment.
DO NOT do it like this: 'This is wrong. Rewrite it.'
DO it like this: Point out the problem + explain why + suggest a solution."

Combining positive and negative

For tasks where style and tone matter a lot, combine both:

"Write the error text for the failed login modal.

Examples of tone I WANT:
- 'Email or password incorrect. Try again.'
- 'Could not log in. Check your details.'

Examples of tone I DO NOT WANT:
- 'ERROR: invalid credentials' (too technical)
- 'Oops! Something went wrong :(' (too informal)"

Key Points 2.4

  • Examples communicate quality expectations better than abstract descriptions
  • Positive examples show the desired pattern
  • Negative examples eliminate unwanted patterns
  • Combining both is the most precise approach for subjective tasks

Topic 2.5 — Iteration: The Art of Refining

Iteration is not a sign of failure

Many beginners get frustrated when the agent's first response is not perfect. This is a misconception.

Iteration is the natural and expected process of working with the agent. Experienced developers do not expect perfection on the first message — they expect a solid base to refine.

Think of the process as sculpture: the first response is the rough marble block. The subsequent messages are the chisels that reveal the final work.

Iteration patterns

Pattern 1 — Refine the scope

[Agent response: long and detailed text]

You: "Good start. Now reduce to at most
       3 paragraphs, keeping only the
       most important points."

Pattern 2 — Adjust the tone

[Agent response: technical and formal text]

You: "The content is right, but the tone is
       too formal. Rewrite it more
       directly, as if explaining
       to a teammate."

Pattern 3 — Ask for alternatives

You: "I liked the structure, but I am not
       sure about the title. Give me 3 alternative
       title options for me to choose from."

Pattern 4 — Expand a section

You: "The third topic is too shallow.
       Expand only that one with more details and
       a practical example."

Pattern 5 — Fix a specific error

You: "Almost perfect. Just one adjustment: on line 3,
       change 'will be able' to 'can' — we want
       certainty, not possibility."

What to do when iteration does not resolve

If after 3-4 iterations you still have not reached the desired result, it is usually a sign of one of these problems:

  1. The initial prompt was too vague — start a new conversation with more context
  2. The task is too complex for a single prompt — break it into subtasks
  3. You do not know exactly what you want — define the objective better before continuing

Dr. Reid's Tip: When stuck, stop and ask the agent: "What would you need from me to do this better?". Sometimes the agent itself points out the missing context.

Key Points 2.5

  • Iteration is the normal process — not a sign of failure
  • There are 5 common patterns: scope, tone, alternatives, expansion and targeted correction
  • After 3-4 iterations without result: restart with more context or break the task

Topic 2.7 — Common Pitfalls (and How to Avoid Them)

Pitfall 1 — The novel-length prompt

Very long prompts filled with irrelevant details confuse more than they help. The agent tries to satisfy all the instructions and may get lost.

❌ [prompt with 20 simultaneous instructions about formatting,
    tone, content, size, audience, project history,
    personal preferences and legal requirements]

✅ Split into 2-3 sequential prompts, focused on one objective
   at a time

Pitfall 2 — Accepting the first response without reviewing

The agent sometimes makes subtle errors — especially in code. Never copy and paste a response without reading it critically.

Develop the habit of checking:

  • Does the code do what was asked?
  • Is there strange logic or excessive simplifications?
  • Do the names of variables, functions and imports make sense?

Pitfall 3 — Ignoring accumulated context

In a long conversation, old messages remain in the context. If you changed your mind about an instruction, make it clear:

✅ "Ignore the formatting instruction I gave earlier.
    From here on, use simple Markdown."

Pitfall 4 — Asking for multiple unrelated tasks in one prompt

❌ "Review this code, write documentation, create 5 tests
    and suggest a better name for the function."

✅ Send one instruction at a time. Finish one, then move
   to the next — or open separate conversations.

Pitfall 5 — Not specifying the language or framework

The agent may assume the wrong language if you do not specify:

❌ "Write a function that sorts a list."
✅ "Write a function in Python 3 that sorts a list
    of dictionaries by the 'date' field, in descending order."

Key Points 2.7

  • Very long prompts or with multiple simultaneous objectives confuse the agent
  • Always review critically — never copy and paste without reading
  • Specify language, framework and version whenever relevant
  • If you change your mind mid-conversation, explicitly declare the change

## Topic 3.4 — Editing Existing Files with the Agent

### Three modes of assisted editing

Editing files with Claude Code is different from creating from scratch. The agent needs to understand the current state of the file to suggest precise modifications.

**Mode 1 — Snippet replacement**

You provide the current snippet and ask for the modified version:

I have this Python function:

def formatar_cpf(cpf):
    return cpf

Modify it to:

  1. Remove non-numeric characters
  2. Add the XXX.XXX.XXX-XX formatting
  3. Validate that it has 11 digits — return None if invalid

The agent responds with the modified function. You replace it in the file.

**Mode 2 — Add to existing file**

When you need to add something without altering what already exists:

Here is my Express routes file [paste the content]. Add at the end a DELETE /usuarios/:id route that:

  • Checks if the ID exists in the database
  • Returns 404 if not found
  • Deletes and returns 200 with confirmation message
  • Maintains the existing code pattern in the file

**Mode 3 — Full refactoring**

For larger refactorings, send the entire file and describe the transformation:

Here is my authentication module [paste the file]. Refactor it to:

  • Replace callbacks with async/await
  • Extract the validation logic into a separate function
  • Maintain the same public interface (function names and parameters)

### The review rule before applying

Before replacing any content in the actual file, always:

1. Read the agent's complete response
2. Verify that the logic is correct
3. Check that the code pattern is consistent with the rest of the project
4. Only then replace in the file

> **Dr. Reid's Tip:** Treat agent responses as suggestions from a senior colleague — you review before approving, not copy blindly. This is not distrust, it is professional responsibility.

### Using version history as a safety net

Before editing important files, use Git to have a rollback point:

```bash
git add .
git commit -m "backup before refactoring with Claude Code"

That way, if the assisted edit does not turn out as expected, you have a way to go back.

Key Points 3.4

  • Three editing modes: snippet replacement, addition, and full refactoring
  • Always provide the current state of the file — the agent does not guess what is already there
  • Review critically before applying any modification
  • Use Git as a safety point before large edits

Topic 3.6 — Working with Multiple Files

The multi-file context challenge

Real projects rarely have problems in a single isolated file. A function in utils.js may be called in controller.js, which imports types from models.js. To reason about real problems, the agent frequently needs more than one file.

Strategies for multi-file context

Strategy 1 — Sequential context

Send the relevant files one at a time, building up context:

Mensagem 1: "Vou te enviar 3 arquivos relacionados a um bug.
             Wait until I send them all."

Message 2: "File 1 — models/usuario.js: [content]"

Message 3: "File 2 — controllers/auth.js: [content]"

Message 4: "File 3 — middleware/validacao.js: [content]"

Message 5: "Based on the 3 files above, identify why
             email validation is passing even for
             emails with invalid format."

Strategy 2 — Project map

For the agent to understand the structure without needing all files, provide a map:

Meu projeto tem a seguinte estrutura:

src/
├── controllers/
│   ├── authController.js  (gerencia login, logout, refresh)
│   └── userController.js  (CRUD of users)
├── models/
│   └── User.js            (modelo Mongoose)
├── middleware/
│   └── auth.js            (valida JWT nas rotas protegidas)
└── routes/
    └── index.js           (registro de todas as rotas)

The problem is in the integration between middleware/auth.js
and authController.js. Can I send you both?

Strategy 3 — Dependency summary

For architecture questions, a summary is more efficient than the complete files:

Tenho um bug de performance. Minha rota GET /relatorio
takes 30 seconds to respond. The flow is:

1. routes/index.js → chama reportController.getRelatorio()
2. reportController.js → chama reportService.generate()
3. reportService.js → faz 3 queries no banco (PostgreSQL)
4. As queries usam JOINs em tabelas de 100k+ registros

Without seeing the code yet, what are the most likely causes
e o que eu deveria investigar primeiro?

Situation Action
File context still relevant Continue in the same conversation
Switched file/feature Open new conversation
Conversation became too long and confusing Open new conversation with summarized context

Key Points 3.6

  • Real problems involve multiple files — send them sequentially
  • Use a structure map to guide the agent without needing to send everything
  • Dependency summaries are more efficient than complete files for architecture questions
  • Start a new conversation when accumulated context is hindering more than helping

Topic 4.1 — Guided Refactoring

What refactoring is (and what it is not)

Refactoring is the practice of improving the internal structure of code without changing its external behavior. It is not rewriting. It is not adding features. It is making code more readable, more maintainable, more efficient — without anything "breaking" for those who use it.

Refactoring examples:

  • Extract a long function into smaller functions
  • Rename variables to more descriptive names
  • Replace duplicated code with an abstraction
  • Convert nested callbacks to async/await
  • Apply a design pattern

Why use the agent to refactor

Manual refactoring is laborious and risky. It is easy to break something unintentionally. The agent speeds up the process in two ways:

  1. Identifies opportunities you may have missed
  2. Executes transformations that would be repetitive to do manually

  • The "What to Avoid" section is the most powerful for preventing rework
  • Document the "why" of decisions — not just the "what"
  • Not every project needs all sections — use the relevant ones

Topic 5.3 — Writing Good Instructions in CLAUDE.md

The actionable instruction principle

An instruction in CLAUDE.md is only useful if it is specific enough for the agent to know how to follow it and testable enough for you to know if it was followed.

❌ "Write clean code."
   (What is "clean code"? Not actionable.)

✅ "Limit functions to a maximum of 20 lines.
    If exceeded, extract to helper functions."
   (Clear, specific, verifiable.)

❌ "Use good security practices."
   (Too vague — which practices?)

✅ "Never concatenate user input in SQL queries.
    Always use prepared statements.
    Never expose stack traces in error responses to the client."
   (Actionable in each case.)

Instruction tone: directive vs. explanatory

For simple and non-obvious conventions, use the directive tone:

- Use semicolons at the end of all statements.
- Prefer `===` over `==` in all comparisons.

For decisions that may seem strange without context, use the explanatory tone:

- Do not use `async/await` in validation middlewares.
  (Reason: our version of Express has a bug with async
  in sequential middlewares — fixing it is in the backlog.)

The explanatory tone prevents the agent from "fixing" an intentional decision by thinking it is an error.

Specificity by area

For code style:

## Style Guide
- Single quotes in JavaScript strings
- Template literals for strings with interpolation
- Destructuring when accessing 2+ properties of an object
- Spread operator instead of Object.assign()
- Arrow functions for callbacks, function declarations for named functions

For error handling:

## Errors
- All errors reach the client as JSON:
  { "error": "friendly message", "code": "INTERNAL_CODE" }
- Never expose internal database error messages to the client
- Validation errors return 400
- Authentication errors return 401
- Authorization errors return 403
- Resource not found errors return 404
- Internal errors return 500 with detailed log on the server

For database:

## Database
- All database operations live in models — never in services or controllers
- Transactions required for operations involving multiple tables
- Migrations in the db/migrations/ folder — never alter schema directly
- Indexes documented with a comment explaining the reason

Key Points 5.3

  • Instructions must be specific and testable — not generic
  • Use directive tone for simple conventions; explanatory for decisions that seem strange
  • Specificity by area (style, errors, database) results in more precise responses

Topic 5.4 — CLAUDE.md in Team Projects

CLAUDE.md as a living document

In teams, CLAUDE.md is not written once and forgotten — it evolves with the project. Treat it like any other code file: with versioning, review, and updates.

Who maintains CLAUDE.md

Model 1 — Tech lead responsibility

The tech lead is the guardian of CLAUDE.md. Each new architecture decision is documented by them. Code review conventions are added after team agreement.

Model 2 — Shared responsibility

Any developer can propose additions via PR. A reviewer approves before merging into the main branch — the same rules as code.

Model 3 — Post-sprint update

At the end of each sprint, the team reserves 15 minutes to review: what changed in the project? What does CLAUDE.md need to reflect?

What to add after team decisions

Every time one of these situations occurs, consider updating CLAUDE.md:

✅ A new library was added to the project
✅ A convention was agreed upon in code review
✅ An architecture decision was made
✅ A problematic pattern was identified and banned
✅ The development environment changed
✅ A new team member joined
✅ A part of the system was significantly refactored

Team-specific sections

Agent onboarding section:

## For the Agent

You are working on a team project with 4 developers.
Priorities in order:
1. Correctness — the code must work correctly
2. Security — especially on endpoints that receive external data
3. Readability — other devs will read and maintain this code
4. Performance — optimize only when there is evidence of a problem

When in doubt about an implementation decision,
present the options and trade-offs instead of choosing alone.

Key Points 5.4

  • CLAUDE.md is a living document — update after each team decision
  • Version it in the repository so all developers benefit
  • Clearly define who is responsible for maintaining it
  • Update whenever: new library, new convention, new architecture decision

Topic 5.5 — Long Context: Strategies for Complex Projects

The context challenge in large projects

Real projects grow. Over time, a single CLAUDE.md may not be sufficient to capture all complexity. Also, the context of a single conversation has limits — in projects with many files, you need strategies to keep context relevant.

Strategy 1 — Modular CLAUDE.md

For projects with multiple distinct domains, you can have CLAUDE.md in subfolders:

project/
├── CLAUDE.md                    ← general project context
├── src/
│   ├── api/
│   │   └── CLAUDE.md            ← API-specific conventions
│   ├── worker/
│   │   └── CLAUDE.md            ← worker-specific conventions
│   └── shared/
│       └── CLAUDE.md            ← what is shared

Claude Code reads the CLAUDE.md closest to the file you are working on, combined with the root one.

Strategy 2 — Session summary

For long conversations, maintain an explicit "current state":

## Current Session State (update as work progresses)

✅ Implemented: Ticket model, basic controller
🔄 In progress: authentication middleware
⏳ Next: controller tests
❌ Blocked: email integration (awaiting credentials)

You update this manually (or ask the agent to update) as you progress.

Strategy 3 — Context by domain

When working on a specific area of the system, open a new conversation and provide focused context:

I will work on the reports feature today.
Relevant domain:
- Model: src/models/Report.js [paste the content]
- Service: src/services/reportService.js [paste the content]
- Report conventions in CLAUDE.md: [paste the relevant section]

Today's task: add filter by date period.

Strategy 4 — Project glossary

For projects with specific business terminology, a glossary section in CLAUDE.md avoids misunderstandings:

## Domain Glossary

- **Ticket:** A technical support request opened by a collaborator
- **Technician:** A member of the IT team who resolves tickets
- **SLA:** Service Level Agreement — maximum resolution time per category
- **Triage:** Process of classifying and prioritizing newly opened tickets
- **Escalation:** Transfer a ticket to a more specialized technician

Key Points 5.5

  • Large projects can have CLAUDE.md in subfolders for specific contexts
  • Session summaries help maintain work state across long conversations
  • Context by domain: open new conversation with focused context when changing areas
  • Domain glossary avoids misunderstandings with specific business terminology

Topic 5.6 — Reusing Context Between Projects

The reinvention problem

If you work on multiple projects, you probably repeat the same personal preferences in each CLAUDE.md: your favorite naming conventions, your commenting style, your error handling preferences.

Your personal CLAUDE.md

You can create a "personal template" CLAUDE.md with the preferences you carry to all projects — and then adapt it for each new project:

# Personal Template — [Your Name]

## My Global Preferences

### Code
- Small functions: maximum 20 lines
- Descriptive names: I prefer verbosity over abbreviations
- Comments: only for non-obvious logic, never for the obvious
- Errors: always explicit, never silent

### Communication
- When in doubt, ask — do not assume
- Present alternatives when there are trade-offs
- Explain the "why" when suggesting a different approach from mine

CLAUDE.md patterns by project type

For Node.js/Express projects:

## Node.js/Express Standard
- Thin routes: only req/res, delegate everything to services
- Middleware for validation, never in the controller
- Environment variables via process.env, documented in .env.example
- Never version .env

For Python/Django projects:

## Python/Django Standard
- PEP 8 for code style
- Docstrings in all public functions
- Settings separated by environment (settings/dev.py, settings/prod.py)
- Migrations always via manage.py makemigrations

For data/analysis projects:

## Data Science Standard
- Notebooks for exploration, .py scripts for production
- Each notebook has a configuration cell at the beginning
- Fixed seed for random operations (reproducibility)
- Raw data never modified — always generate derivatives

Key Points 5.6

  • Create a personal CLAUDE.md template with your global preferences
  • Adapt the template for each new project instead of starting from scratch
  • Patterns by project type (Node, Python, data) speed up initial setup

Topic 6.2 — Writing Personas with Depth

What is an agent persona

A persona is the professional identity assigned to the agent for a task or set of tasks. It is not fantasy — it is a way to activate the most appropriate reasoning for the context.

When you say "act as a senior code reviewer", the agent automatically calibrates:

  • The level of rigor of the analysis
  • The criteria it uses to evaluate
  • The appropriate technical vocabulary
  • The tone of the feedback

The 4 depth levels of persona

Level 1 — Title only (weak)

"Act as a developer."

Too vague. What type of developer? With what specialty? For what context?

Level 2 — Title + specialty (basic)

"Act as a backend developer specializing in Node.js."

Better, but still missing the work context and criteria.

Level 3 — Title + specialty + work context (good)

"Act as a senior backend developer specializing in Node.js,
with experience in high-availability APIs for the financial sector."

Already calibrates the rigor level and knowledge domain.

Level 4 — Complete: identity + mission + criteria + tone (excellent)

"Act as a senior software architect with 15 years of experience
in distributed systems for fintechs.

Your mission: review code before it goes to production,
identifying not only what is wrong today, but what will
create problems when volume grows 10x.

You are demanding, but constructive. Never say just 'this is wrong'
— always explain the impact and suggest the solution.

Your evaluation criteria, in priority order:
1. Correctness and security
2. Scalability
3. Maintainability
4. Performance"

Building each persona component

Identity — who is this agent:

Elements:
- Professional role/title
- Years or level of experience
- Technical specialty
- Industry context (if relevant)

Examples:
"senior security engineer with focus on OWASP"
"corporate tech writer specializing in API documentation"
"data architect with experience in legacy system migration"

Mission — what it exists for:

Elements:
- Action verb (review, create, diagnose, train...)
- Work object (code, documentation, architecture...)
- Final objective (find X before Y, ensure Z...)

Examples:
"Your mission is to find bugs that automated tests miss"
"You exist to transform technical code into documentation
 that any stakeholder can understand"
"Your job is to identify technical debt before it
 becomes a fire"

Criteria — how it evaluates:

Elements:
- What it prioritizes (order matters)
- What it never ignores
- The level of rigor

Examples:
"Priorities: security > correctness > performance > aesthetics"
"You never approve code without explicit error handling"
"Your minimum standard is code that passes senior review"

Tone — how it communicates:

Elements:
- Directiveness (direct vs. diplomatic)
- Style (technical vs. accessible)
- Posture (collaborative vs. critical)

Examples:
"Direct and objective — no beating around the bush, no softening feedback"
"Technical but accessible — uses analogies when needed"
"Always constructive — points out problem + explains impact + suggests solution"

Key Points 6.2

  • Personas have 4 depth levels — Level 4 produces substantially better results
  • The 4 components: identity, mission, criteria, tone
  • Build each component with specific, not abstract, elements

Topic 6.3 — Scope and Restrictions

Why delimiting scope is so important

An agent without a clear scope tends to:

  • Do more than was asked (scope creep)
  • Comment on things outside its designated competency
  • Mix different priorities

An agent with a well-defined scope:

  • Focuses on what matters for that task
  • Recognizes when something is outside its scope and reports it
  • Maintains consistency between sessions

Defining the scope

Task scope — what it does:

"You analyze JavaScript and TypeScript code.
 Your analyses cover: security, performance, readability, and tests.
 You do NOT do: system architecture, product decisions,
 deadline estimates."

Domain scope — what area it specializes in:

"Your specialty is authentication and authorization in REST APIs.
 Outside this scope, you can give general opinions but signal
 that it is not your main area of expertise."

Output scope — what it delivers:

"Your output always includes:
 1. List of problems found (with severity)
 2. Explanation of the impact of each
 3. Fix suggestion with code
 You do not deliver: long introductory text, unnecessary summaries,
 opinions on product decisions."

Effective restrictions

Restrictions are what the agent never does — regardless of what is asked. They are the "never ultrapassar" do escopo.

Behavioral restrictions:

"Never approve code that:
 - Has no error handling
 - Exposes sensitive information in logs
 - Has SQL concatenated with user input"

Communication restrictions:

"Never say just 'it is wrong' without explaining why.
 Never suggest a fix without showing the code.
 Never use terms like 'simple', 'obvious' or 'trivial'
 — this can demotivate junior developers."

Scope restrictions:

"If asked for something outside the scope of code review
 (such as deadline estimates or product decisions),
 politely acknowledge and redirect to the focus."

The "if X then Y" technique

For conditional restrictions, use the "if/then" structure:

"If the code has a critical security vulnerability,
 flag it as BLOCKER before any other analysis.
 The review cannot be approved with open blockers."

"If the code is well written and has no relevant issues,
 say that clearly. Do not invent problems to seem
 more thorough."

"If unsure about an architecture decision,
 present the options with trade-offs — do not choose for me."

Key Points 6.3

  • Scope defines what the agent does; restrictions define what it never does
  • Delimit: task scope, domain scope, and output scope
  • Use the "if X then Y" technique for conditional restrictions
  • An agent without scope becomes a generalist — loss of specialization and consistency

Topic 6.6 — Pattern: Debugging Agent

Complete template

Act as a debugging and system diagnosis specialist.

IDENTITY:
You are the type of engineer teams call when no one
else can figure out what is happening. Your specialty
is systematic reasoning — you never "guess" a solution,
you build hypotheses and test them methodically.

MISSION:
Find the ROOT CAUSE of the problem — do not mask the symptom.
A fix that hides the bug is worse than no fix.

PROCESS:
1. UNDERSTANDING: Make sure you understand exactly the symptom
   before raising hypotheses. If you need more information, ask.
2. HYPOTHESES: List possible causes in order of probability
3. ISOLATION: Suggest how to test each hypothesis with minimum effort
4. ROOT CAUSE: Identify the real problem before suggesting the fix
5. FIX: Provide the fix with an explanation of why it works
6. PREVENTION: Suggest how to avoid this type of problem in the future

FORMAT:
## Problem Understanding
[confirmation or request for clarification]

## Hypotheses (by probability)
1. [most likely] — [why]
2. [second hypothesis] — [why]

## How to Test
[steps to confirm each hypothesis]

## Root Cause
[when confirmed]

## Fix
[corrected code with explanation]

## Prevention
[how to avoid in the future]

RESTRICTIONS:
- Never suggest a fix without identifying the root cause first
- If lacking sufficient information, ASK before raising hypotheses
- Never ignore the "but it works on my machine" — investigate the difference
- If the problem may have multiple simultaneous causes, mention all of them

Topic 6.7 — Pattern: Code Mentoring Agent

Complete template

IDENTITY:
You combine technical depth with pedagogical skill.
You know that how you explain is as important as
what you explain. You have been in the junior developer's
place — and have not forgotten what it is like to not understand something.

MISSION:
Not just to solve the problem — to teach the reasoning behind
the solution. Your success is measured when the developer can
solve the next problem alone, not when they copied your solution.

PEDAGOGICAL STYLE:
- Ask before explaining: "What did you try? What do you think
  is happening?" — activate reasoning before giving the answer
- Build on what the student already knows
- Use real-world analogies for abstract concepts
- Praise what is right before pointing out what to improve
- Explain the "why" before the "how"

SUPPORT LEVELS:
- HINT: For problems where the student is close — give a direction without revealing
- GUIDE: For problems needing more support — guide step by step
- SOLUTION: For blocking problems — explain completely with context

RESTRICTIONS:
- Never give the complete solution if the student is close to getting there alone
- Never use "simple", "easy" or "obvious" — this demotivates
- Never ignore the code the student already wrote — start from what exists
- Always end with a question or reflection that stimulates the next step

Module 7 — Orchestrating Agents

[Module 7 of 8] ██████████████░░

Learning Objective: By the end of this module, you will be able to identify when multiple agents are more effective than a single one, apply the three orchestration patterns (sequential, parallel, and hierarchical), pass context between agents precisely, and build functional multi-agent pipelines without paid tools.

Estimated duration: 4 hours Level: Intermediate+ Prerequisites: Modules 1 through 6 completed. You know how to create specialized agents with deep personas.

Why orchestration is the next level

In Module 6, you learned to create specialized agents — each excellent at one thing. But complex tasks rarely fit into a single specialty.

Think about how real projects work: a requirement arrives, is refined by an analyst, implemented by a developer, reviewed by another, tested by a QA and documented by someone. Each step uses a different perspective, and the result of each step feeds the next.

Agent orchestration replicates this flow in Claude Code. Instead of asking everything from a single generic agent, you define a chain where each agent does what it does best — and passes the product of its work to the next.

The result: workflows that consistently produce superior quality, in fewer iterations, with context preservado ao longo do processo.

Topic 7.1 — Solo Agent vs. Multiple Agents

When a single agent is sufficient

Do not use multiple agents when you do not need to. Orchestration adds complexity — it is worth it only when the gainho compensa.

Use a single agent when:

✅ The task is well-delimited and has a single objective
✅ One specialty completely covers what you need
✅ The result will not be input for another task
✅ The task can be completed in 2-3 iterations
✅ The necessary context fits in a single conversation

Examples:
- Fix a specific bug
- Generate JSDoc documentation for a file
- Review an isolated function
- Answer a technical question

Use multiple agents when:

✅ The task has distinct phases with different criteria
✅ Different phases require different specialties
✅ The output of one phase needs to be evaluated from another perspective
✅ Genuine parallelism can speed up independent tasks
✅ A "supervisor" agent needs to coordinate specialized work

Examples:
- Implement a feature + review the code + document
- Analyze requirements + implement + test
- Generate draft + review tone + adapt for audience
- Debug + fix + write regression test

The cost of orchestration

Orchestration has real costs you need to consider:

Cost Description
Time Multiple conversations take more time than a single one
Context Each agent starts without the history of previous ones
Consistency You need to ensure context was transferred correctly
Cognitive overhead You coordinate the transitions — this requires attention

Orchestration is worth it when the quality gain outweighs these costs. For simple tasks, it is not worth it.

Key Points 7.1

  • Use solo agent for simple and well-delimited tasks
  • Use multiple agents when there are distinct phases with different specialties
  • Orchestration has real cost — use only when the gain is worth it

Pattern 1 — Sequential (the most common)

What it is: Agents work in series — the output of Agent A becomes the input of Agent B, which becomes the input do Agente C.

When to use: When tasks have order dependency — each step needs the result of the previour.

[Agent C: Documentation]
        ↓

→ Receives: final code from Agent 2

What it is: Multiple agents work simultaneously on independent parts of the same problem, and the results are combined.

When to use: When there are independent subtasks that can be done at the same time without depending on eachs das outras.

Topic 7.3 — Context Transfer Between Agents

The central problem of manual orchestration

When you use multiple agents in Claude Code, each new conversation starts from scratch. Agent B knows nothing about what Agent A did — unless you transfer that context explicitly.

This context transfer is the heart of orchestration. Doing it well is what separates pipelines that funcionam de pipelines que produzem resultados inconsistentes.

The 3 elements of context to transfer

1. The result of the previous agent

Always transfer the literal output of the previous agent — not a summary or your interpretation. The next agent needs do material real para trabalhar.

❌ "Agent 1 said the code was ok with some issues."
✅ [paste the complete output of Agent 1]

2. The continuity instruction

The second agent needs to know:

  • What the previous agent did
  • What you expect from the current agent
  • What the state of the work is at this point
✅ "Agent 1 (implementer) just created the code below.
    You are Agent 2 (security reviewer).
    Review the code considering: [criteria]

    Code from Agent 1:
    [output of Agent 1]"

3. The project context (if not in CLAUDE.md)

If the agent does not have access to the project CLAUDE.md, provide the relevant context explicitly:

✅ "Project context:
    - Node.js/Express REST API for ticketing system
    - PostgreSQL without ORM
    - JWT with two roles: colaborador and rh_admin

    Agent 1 implemented the endpoint below. Review:"

Topic 7.7 — Orchestration Pitfalls

Pitfall 1 — Context lost in transfer

Symptom: Agent B ignores or contradicts what Agent A did.

Cause: The output of Agent A was transferred in a summarized or incomplete way.

Solution: Always paste the complete and literal output. Use the transfer template. Never paraphrase.

Pitfall 2 — Pipeline too long

Symptom: The pipeline has 5+ steps, each depending on the previous, and you lose track after 2 hours.

Cause: Excessive decomposition of a task that could be done in fewer steps.

Solution: Question each step: "does this really require a different agent, or can the previous one handle it?". Reduce to the minimum necessary.

Pitfall 3 — Mixing agents in the same conversation

Symptom: A conversation starts with the Implementer Agent and you ask for a security review in the midd, sem nova conversa.

Cause: Convenience — it seems faster.

Consequence: The agent tries to be implementer and reviewer at the same time, losing the specialization ofe ambos.

Solution: Always open a new conversation for each agent. Separation is what guarantees specialization.

Pitfall 4 — Inconsistent output between steps

Symptom: Agent 2 receives Node.js code but starts suggesting Python because it had no context about stack.

Cause: The project context was not transferred along with the output.

Topic 8.1 — What is the Model Context Protocol (MCP)

The core idea

MCP is an open protocol created by Anthropic that defines a standardized way to connect language models to external tools and data sources.

Think of it this way: if Claude Code is your assistant, MCPs are the tools on the workbench that assistant can use. Without MCPs, it responds with text. With MCPs, it can:

  • Read and write files on your operating system
  • Query repositories on GitHub
  • Execute queries on local databases
  • Make HTTP requests to external APIs
  • Access Git history
  • And much more

How MCP works in practice

┌────────────────────────────────────────────────────┐
│  BEFORE MCP                                        │
│                                                    │
│  You  →  copy content  →  Claude Code             │
│  Claude Code  →  responds with text  →  You        │
│  You  →  apply manually                           │
└────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────┐
│  WITH MCP                                          │
│                                                    │
│  You  →  natural instruction  →  Claude Code       │
│  Claude Code  →  uses MCP  →  external system      │
│  external system  →  result  →  Claude Code        │
│  Claude Code  →  responds with real context        │
└────────────────────────────────────────────────────┘

The protocol is open

One of the most important characteristics of MCP is being an open standard. This means:

  • Any developer can create an MCP
  • The community has already created hundreds of MCPs for the most varied systems
  • Free MCPs cover the vast majority of everyday use cases
  • You do not need any paid service to use useful MCPs

MCP vs. VS Code extensions

It is important to distinguish:

MCP VS Code Extension
What it is Server that exposes tools to Claude Code Plugin that adds features to the editor
Who uses it Claude Code (via protocol) VS Code (directly)

Runs As a separate process (local server) Inside VS Code

Topic 8.2 — Installing and Configuring MCPs in VS Code

Where MCP settings are stored

Claude Code MCP settings are stored in an extension configuration JSON file. The location varies by operating system:

Windows:

%APPDATA%\Claude\claude_desktop_config.json

or, depending on the extension version:

%USERPROFILE%\.claude\settings.json

macOS:

~/Library/Application Support/Claude/claude_desktop_config.json

Linux:

~/.config/claude/claude_desktop_config.json

How to find it: In VS Code, press Ctrl+Shift+P and search for "Claude: Open Settings" or "Claude: Configure MCP". If it does not appear, access the Claude Code panel > settings icon.

Topic 8.6 — Security and Best Practices with MCPs

The principle of least privilege

Just as in security systems, configure MCPs with the minimum necessary access:

✅ Filesystem: point only to the projects folder
✅ GitHub: use minimum scopes on PAT (public_repo if possible)
✅ SQLite: point only to the specific database

❌ Do not point Filesystem to / (root) or C:\
❌ Do not use tokens with all scopes for convenience
❌ Do not configure SQLite for production databases

Protecting tokens and credentials

Absolute rule: Never commit the MCP configuration file to the repository.

The file stays in the user directory (~/.claude/ or %APPDATA%\Claude\) — outside any repository. This is intentional.

If you need to share the configuration with the team:

✅ Create a .mcp-config.example.json file without real tokens
✅ Add .mcp-config.json to .gitignore if you need to version it
✅ Document in the README which MCPs the project uses and how to configure
✅ Use operating system environment variables when possible

Reviewing what the agent did with MCPs

When the agent uses an MCP to create or modify files, always review:

Before finalizing:
□ Is the created/modified file the correct one?
□ Is the content correct and safe?
□ Was no sensitive data exposed?
□ Was the file created in the correct location?

Dr. Reid's Tip: MCPs that write files or create issues are more powerful — and therefore require more attention. It is different from the agent writing text that you review before saving. With the Filesystem MCP, the file is already on disk. With the GitHub MCP, the issue has already been created. Review your instructions carefully before sending.

Key Points 8.6

  • Apply the principle of least privilege — minimum necessary access
  • Never commit tokens to the repository — the configuration file stays in the user directory
  • Document the MCPs used in the project to make team setup easier
  • MCPs that write/create require more attention — the effect is immediate
